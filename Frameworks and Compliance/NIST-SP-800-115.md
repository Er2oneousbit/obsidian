# NIST SP 800-115

#NIST #SP800115 #PenTest #Methodology #Testing

## What is this?

**NIST Special Publication 800-115** — Technical Security Testing and Assessment Procedures. Detailed guidance for security professionals conducting technical testing: vulnerability assessments, penetration testing, security code review, malware testing. Published 2008; still the authoritative technical testing standard for federal systems and contractors.

---

## Overview

**NIST SP 800-115 Basics:**
- **Purpose**: Standardize how technical testing is conducted; provides detailed procedures.
- **Audience**: Federal agencies, contractors, testers, code reviewers.
- **Scope**: Technical testing methodologies; how to execute each test type.
- **Length**: ~300 pages (very detailed; textbook-like).

**vs. Related Standards**:
- **PTES** = overall pentest framework (phases, engagement); lighter on technical detail.
- **NIST SP 800-115** = detailed technical testing procedures; very prescriptive.
- **OWASP Testing Guide** = web app testing checklist; similar detail level but web-focused.

**When Used**: Federal contracts require 800-115 compliance; insurance/compliance audits; organizations wanting rigorous testing procedures.

## NIST SP 800-115 Structure

### Part 1: Security Testing Fundamentals

#### What is Security Testing?

**Definition**: Systematic assessment of systems to find vulnerabilities and weaknesses.

**Goals**:
- Identify vulnerabilities before adversaries do.
- Measure compliance with security standards (NIST SP 800-53, CIS Controls, etc.).
- Validate effectiveness of security controls.
- Support risk management decision-making.

#### Types of Testing

| Test Type | Definition | Goal |
|---|---|---|
| **Vulnerability Assessment** | Identify known vulnerabilities using automated tools | Find low-hanging fruit; quick baseline |
| **Penetration Testing** | Active exploitation; demonstrate real-world impact | Prove impact; show what attackers can do |
| **Security Code Review** | Manual/automated analysis of source code | Find logic flaws, crypto issues, input validation bugs |
| **Malware Analysis** | Reverse-engineering malware; understand behavior | Know what malicious code does; improve detection |
| **Wireless Assessment** | Test WiFi security | Weak encryption, rogue APs, credential theft |

#### Phases of 800-115 Testing

800-115 defines a **4-phase** assessment methodology (the Attack phase loops back to Discovery when new hosts or data are found):

1. **Planning**: Define objectives, scope, constraints, ROE (Rules of Engagement).
2. **Discovery**: Information gathering (passive + active recon) **and** vulnerability analysis/scanning.
3. **Attack**: Verify and exploit vulnerabilities to demonstrate impact; loops back to Discovery for newly discovered systems.
4. **Reporting**: Document findings and remediation guidance (runs throughout, finalized at the end).

### Part 2: Vulnerability Assessment (Automated Testing)

**Goal**: Use automated tools to quickly identify known vulnerabilities.

#### Network Vulnerability Assessment

**Tools**: Nessus, Qualys, OpenVAS, Rapid7 Nexpose.

**Process**:

1. **Define Scope**:
   - Which IP ranges/subnets to scan?
   - Exclude production systems, critical infrastructure.
   - Coordinate with network team (scans can impact systems).

2. **Select Assessment Tool**:
   - Consider: Coverage (does it find vulnerabilities you care about?), ease of use, reporting, cost.
   - Nessus = industry standard (expensive, comprehensive).
   - OpenVAS = free, open-source (less comprehensive but adequate).
   - Qualys = cloud-based (good for continuous monitoring).

3. **Configure Scanning Parameters**:
   - **Scan type**: Credentialed (provide admin creds to scanner) vs. non-credentialed (external view).
   - **Plugins**: Enable/disable specific vulnerability checks (e.g., disable fragile checks in production).
   - **Sensitivity**: Conservative (fewer false positives) vs. aggressive (more thorough, higher false positive rate).
   - **Timing**: Aggressive (faster, higher load) vs. stealthy (slower, less noticeable).

4. **Execute Scan**:
   - Schedule off-peak hours (don't impact business).
   - Monitor for issues (if system gets overwhelmed, stop scanning).

5. **Analyze Results**:
   - Vulnerability severity (CVSS score).
   - False positives (tool thinks something is wrong when it's not).
   - Context (is vulnerability actually exploitable in your environment?).

6. **Report Findings**:
   - List all vulnerabilities (with CVSS scores, CVSS vectors).
   - Categorize by severity (Critical, High, Medium, Low).
   - Note false positives.
   - Provide remediation steps (patch version, configuration change, etc.).

#### Application Vulnerability Assessment

**Tools**: Burp Suite, OWASP ZAP, Qualys WEBASCAN, Rapid7 AppSpider.

**Process** (similar to network assessment):

1. **Scope**: Which web applications/APIs to test? (scope creep risk: 100s of endpoints).

2. **Configure Tool**:
   - Target URLs.
   - Authentication (if app requires login).
   - Scan scope (which pages/parameters to test).
   - Attack surface (forms, headers, cookies, API endpoints?).

3. **Execute Scan**:
   - Tool crawls app (discovers pages, forms).
   - Tests each input for vulnerabilities (SQL injection, XSS, CSRF, etc.).
   - Identifies insecure configurations (missing security headers, weak TLS, etc.).

4. **Analyze Results**:
   - Cross-site scripting (XSS) = high priority.
   - SQL injection = critical.
   - Insecure direct object references (IDOR) = high.
   - Missing CSRF token = medium.
   - Informational findings (server version disclosure) = low.

5. **Report & Remediate**:
   - PoC (proof of concept) for each finding.
   - Remediation steps (input validation, parameterized queries, etc.).

### Part 3: Penetration Testing (Active Exploitation)

**Goal**: Demonstrate real-world impact by exploiting vulnerabilities.

#### Planning Penetration Test

**Pre-Test Activities** (see PTES Phase 1):
- Objectives: What are we trying to prove?
- Scope: What systems are in scope?
- ROE: What are we allowed to do?
- Timeline: When should testing occur?
- Escalation: If we find critical issue, who do we notify immediately?

#### Reconnaissance Phase

**Passive Reconnaissance** (no direct contact):

```bash
# Public WHOIS/DNS info
whois <target.com>
dig <target.com> ANY

# Subdomain enumeration (passive)
subfinder, amass (passive mode)

# Certificate logs (TLS certs often leak subdomains)
curl -s "https://crt.sh/?q=target.com"

# Email/employee enumeration
theHarvester, Hunter.io

# GitHub leaks
gitleaks scan (search for API keys, credentials)
```

**Active Reconnaissance** (direct probing):

```bash
# Network scanning
nmap -sV -O <target> (port scan, service version, OS detection)

# Enumerate services
nmap --script default <target> (default NSE scripts)
nmap -script vuln <target> (vulnerability checks)

# Web app enumeration
ffuf -w wordlist -u http://<target>/FUZZ (directory brute force)
```

**Important**: Reconnaissance is **loud** (lots of traffic); target's IDS will detect it. Coordinate timing.

#### Vulnerability Discovery (Active)

**Approaches**:
1. **Tool-assisted** (faster, more systematic):
   - Run Burp/ZAP automated scan on web app.
   - Run Nessus/Qualys on network.
   - Review tool findings.

2. **Manual** (catches logic flaws tools miss):
   - Manually test each input for injection, encoding, etc.
   - Review code (if available).
   - Test for business logic flaws.

#### Exploitation

**For each vulnerability, tester**:
1. Confirms it's real (not false positive).
2. Develops PoC (proof of concept).
3. Demonstrates impact (shows what attacker can do).
4. Documents impact (screenshots, evidence).

**Example: SQL Injection Exploitation**

```bash
# Vulnerable URL: http://target.com/user.php?id=1

# Test for injection
curl "http://target.com/user.php?id=1' OR '1'='1"
# If vulnerable, returns more data than expected (true condition)

# Enumerate database
sqlmap -u "http://target.com/user.php?id=1" --dbs
# Returns: database1, database2, ...

# Extract data
sqlmap -u "http://target.com/user.php?id=1" -D database1 --tables
sqlmap -u "http://target.com/user.php?id=1" -D database1 -T users --dump
# Returns usernames, passwords, etc.
```

#### Post-Exploitation Testing

If ROE allows:

```bash
# Establish persistence
reverse shell, backdoor account, cron job

# Privilege escalation
sudo misconfiguration, kernel exploit, weak file permissions

# Lateral movement
SSH key reuse, Kerberoasting, pass-the-hash

# Data collection
grep for sensitive files, exfiltrate sample data (not full DB)
```

**Important**: Many ROE explicitly disallow post-exploitation (too risky; can crash systems). Document what's authorized before testing.

### Part 4: Security Code Review

**Goal**: Find vulnerabilities in application source code before deployment.

#### Static Analysis (SAST — Source Code Analysis)

**Tools**: SonarQube, Veracode, Fortify, Checkmarx, FindBugs.

**Process**:

1. **Scan Source Code**:
   - Tools parse code; build abstract syntax tree (AST).
   - Check against security rules (CWE, OWASP).
   - Find vulnerable patterns (hardcoded credentials, SQL injection, weak crypto, etc.).

2. **Analyze Findings**:
   - High-confidence issues (obvious SQL injection, hardcoded password).
   - Low-confidence issues (might be false positive; requires manual review).

3. **Report**:
   - Issue type (SQL injection, XSS, etc.).
   - Location (file, line number).
   - Code snippet.
   - Remediation suggestion.

#### Dynamic Analysis (DAST — Runtime Testing)

**Tools**: Burp Suite, OWASP ZAP (web apps); Valgrind, AddressSanitizer (memory issues).

**Process**:

1. **Instrument Application**:
   - Insert monitoring code to track data flow.
   - Observe behavior at runtime.

2. **Fuzz Inputs**:
   - Send unexpected/malicious inputs.
   - Observe if app crashes, behaves unexpectedly.

3. **Detect Vulnerabilities**:
   - Memory leaks, buffer overflows, null pointer dereference, use-after-free.
   - Format string vulnerabilities, integer overflows.

#### Manual Code Review

**Approach**: Tester manually reads code; looks for logic flaws.

**Focus Areas**:
- **Authentication/Authorization**: Proper access control? Can unauthenticated user access data?
- **Cryptography**: Using strong algorithms? Keys stored securely? No hardcoded secrets?
- **Input Validation**: Is all user input validated? Protected against injection?
- **Error Handling**: Do error messages leak sensitive info?
- **Configuration**: Secrets in code? Hardcoded passwords?
- **Dependency Management**: Using known-vulnerable libraries?

**Example**: Reviewing login function for security flaws

```python
def login(username, password):
    # BAD: SQL injection vulnerability
    query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
    user = db.execute(query)
    
    # BAD: Plaintext password comparison; no rate limiting
    if user and user.password == password:
        return "LOGIN_SUCCESS"
    else:
        return "INVALID_CREDENTIALS"  # No rate limit; allows brute force
```

**Findings**:
1. SQL Injection (CWE-89).
2. Plaintext password storage/comparison (CWE-256).
3. No rate limiting (CWE-307, allows brute force).

### Part 5: Malware Assessment

**Goal**: Understand malware behavior; assess detection/response capabilities.

#### Static Malware Analysis

```bash
# File properties
file malware.exe  # identify file type
strings malware.exe  # extract readable strings (IP addresses, domains, etc.)
md5sum malware.exe  # get hash; check against threat databases (VirusTotal)

# Reverse engineering
IDA Pro (disassembler; convert binary to assembly code)
Ghidra (free disassembler; NSA tool)
objdump (Linux binary analysis)
```

#### Dynamic Malware Analysis

```bash
# Run in sandbox environment (isolated VM)
# Observe behavior:
# - File creation/modification
# - Registry changes (Windows)
# - Network connections (DNS queries, C2 callbacks)
# - Process creation
# - Persistence mechanisms

# Tools: Cuckoo Sandbox, Any.run, Hybrid Analysis
```

#### Assessment & Reporting

- **Capabilities**: What does malware do? (steal credentials, open backdoor, encrypt files?)
- **Infrastructure**: C2 addresses, malware distribution servers.
- **Indicators of Compromise (IoCs)**: File hashes, domains, IPs to block.
- **Detection Methods**: What signatures/rules detect this malware?

### Part 6: Wireless Assessment

**Goal**: Test WiFi security; identify weak encryption, rogue APs, credential theft.

#### Tools & Techniques

```bash
# Identify WiFi networks, signal strength
iwlist scanning  # Linux
netsh wlan show networks  # Windows

# Crack WPA2/WPA3
hashcat, John the Ripper (brute force password)
Aircrack-ng (WiFi cracking tool)

# Rogue AP detection
iwlist scan | grep SSID  # look for duplicate SSIDs
Kismet (passive WiFi monitoring)

# Credential interception
Wireshark (packet capture; see unencrypted traffic)
```

## NIST SP 800-115 Security Testing Best Practices

### Planning Phase

- [ ] Clearly define objectives (what are we testing for?).
- [ ] Define scope (what's in/out of scope).
- [ ] Get written authorization (protect from legal liability).
- [ ] Document ROE (what are we allowed to do?).
- [ ] Schedule testing (coordinate with target to avoid business impact).

### Execution Phase

- [ ] Use baseline configurations for comparison (know what's normal).
- [ ] Test during representative time windows (not just off-hours; weekday testing catches different issues).
- [ ] Document all findings (screenshots, timestamps, evidence).
- [ ] Preserve evidence (don't modify target systems unnecessarily).
- [ ] Report critical findings immediately (don't wait for final report).

### Analysis Phase

- [ ] Validate findings (eliminate false positives).
- [ ] Determine exploitability (is this a real risk, or theoretical?).
- [ ] Assess business impact (what's at risk?).
- [ ] Prioritize remediation (by severity × likelihood).

### Reporting Phase

- [ ] Include executive summary (non-technical, high-level).
- [ ] Document methodology (how testing was performed).
- [ ] Provide clear remediation steps (specific, actionable).
- [ ] Support with evidence (screenshots, PoC code).

## Comparison: Vulnerability Assessment vs. Penetration Testing

| Aspect | Vulnerability Assessment | Penetration Testing |
|---|---|---|
| **Goal** | Find known vulnerabilities | Demonstrate real-world impact |
| **Tools** | Automated scanners (Nessus, ZAP) | Manual exploitation + tools |
| **Depth** | Shallow (breadth) | Deep (specific vulnerabilities) |
| **Time** | Fast (hours) | Slow (days/weeks) |
| **Cost** | Low ($1–$5K) | High ($5–$50K+) |
| **Output** | List of vulnerabilities | Narrative of attack chain; impact |
| **Example Finding** | "SQL Injection on login form" | "Attacker can execute shell commands; steal customer database" |

**Typical Approach**: Run vulnerability assessment first (find obvious issues); follow up with penetration testing (demonstrate impact of selected vulnerabilities).

## Risk Assessment Framework (Post-Testing)

**For each vulnerability**:

**Risk = Likelihood × Impact**

| Factor | Assessment |
|---|---|
| **Likelihood** | How easy is it to exploit? (1–5 scale) |
| **Impact** | What's the business damage? (1–5 scale) |
| **Risk Score** | Likelihood × Impact (1–25) |
| **Priority** | 20–25 = Critical (fix immediately) / 15–19 = High / 10–14 = Medium / <10 = Low |

## Common Testing Mistakes (800-115 Perspective)

| Mistake | Impact | Fix |
|---|---|---|
| **No clear objectives** | Testing unfocused; wastes time | Define upfront (what vulnerabilities matter most?) |
| **False positives** | Team wastes time chasing non-issues | Validate all findings before reporting |
| **Over-reliance on automation** | Miss logic flaws, business logic issues | Combine automated + manual testing |
| **Inadequate documentation** | Can't reproduce findings; client disputes them | Screenshot everything; document steps |
| **Testing only obvious issues** | Miss subtle vulnerabilities | Test systematically (all inputs, all configurations) |
| **No evidence preservation** | Can't prove findings in post-mortem | Take screenshots, save logs, preserve state |
